# 06 — Что НЕ сработало

## 1. Захват системного звука через Web Audio API

**Что пытались**: Использовать Web Audio API в renderer-процессе для захвата системного звука через `navigator.mediaDevices.getUserMedia()`.

**Почему не сработало**: Web Audio API не может захватить системный звук (loopback) — это ограничение безопасности браузера. `getUserMedia()` предоставляет доступ только к микрофону и камере. Для системного звука нужен нативный код.

**Решение**: Переход на WASAPI loopback через нативный модуль `win-audio-capture` в main-процессе, с fallback на ffmpeg child_process.

**Версия**: V100 (дизайн-решение, до реализации)

## 2. Отправка raw PCM на Whisper API

**Что пытались**: Отправлять raw PCM-данные (16kHz Mono 16-bit) напрямую на Whisper API, ожидая что API сам распознает формат.

**Почему не сработало**: Whisper API требует корректный контейнер (WAV, MP3, M4A, WEBM). Raw PCM без заголовка — невалидный аудио-файл, API возвращает ошибку формата.

**Решение**: Функция `createWavBuffer()` в sttService.ts — добавляет 44-байтный WAV-заголовок (RIFF/WAVE/fmt/data) к PCM-данным перед отправкой.

**Версия**: V100

---

## 3. Использование npm-пакета `form-data` для multipart

**Что пытались**: Использовать популярный npm-пакет `form-data` для формирования multipart/form-data запросов к Whisper API.

**Почему не сработало**: Добавляет лишнюю зависимость, а в Electron main-процессе с `externalizeDepsPlugin()` нативные зависимости могут ломать сборку. К тому же `form-data` не всегда корректно работает с Buffer в Electron.

**Решение**: Ручная сборка multipart через `Buffer.concat()` — строки заголовков как Buffer.from(string, 'utf-8'), бинарные данные WAV как есть. Boundary генерируется динамически. Это работает и не требует зависимостей.

**Версия**: V100

## 4. Параллельная отправка чанков на STT

**Что пытались**: Отправлять чанки на Whisper API параллельно (Promise.all или несколько fetch одновременно) для снижения задержки.

**Почему не сработало**: Groq rate limit — 30 запросов/мин на бесплатном плане. Параллельная отправка 2-3 чанков одновременно может исчерпать лимит за секунды. Кроме того, порядок результатов не гарантируется — транскрипции могут прийти в неправильном порядке.

**Решение**: Последовательная обработка очереди (while loop в `processQueue()`). Следующий чанк отправляется только после получения результата предыдущего.

**Версия**: V100

## 5. ffmpeg dshow «Stereo Mix» на всех Windows-машинах

**Что пытались**: Использовать ffmpeg с dshow и устройством `audio=Stereo Mix (Realtek Audio)` как fallback для захвата системного звука на Windows.

**Почему не сработало**: «Stereo Mix» — виртуальное устройство, которое по умолчанию ОТКЛЮЧЕНО в настройках звука Windows. Многие аудиодрайверы (особенно не-Realtek) не предоставляют его вообще. Имя устройства различается на разных машинах (например, «Stereo Mix (High Definition Audio Device)»).

**Решение**: Основной метод — WASAPI loopback через win-audio-capture (не зависит от Stereo Mix). ffmpeg fallback — запасной вариант, но он ненадёжен для системного звука на Windows.

**Версия**: V100

## 6. macOS системный звук через ffmpeg avfoundation

**Что пытались**: Использовать ffmpeg с `avfoundation` и устройством `:BlackHole 2ch` для захвата системного звука на macOS.

**Почему не сработало**: BlackHole — сторонний виртуальный аудио-драйвер, его нужно устанавливать отдельно. Без него avfoundation не может захватить системный звук. Кроме того, на Apple Silicon Mac есть дополнительные ограничения на доступ к аудио-устройствам.

**Решение**: ffmpeg fallback с BlackHole — документирован, но требует ручной установки пользователем. В будущем нужен нативный AudioUnit-based захват для macOS (coreaudio).

**Версия**: V100

## 7. Использование `fetch` с `response.body` в Electron для SSE

**Что пытались**: Использовать нативный `fetch()` в Electron main-процессе для чтения SSE-стрима от OpenRouter.

**Почему не сработало (частично)**: В целом это работает, но есть подводные камни:
- `response.body?.getReader()` может вернуть null если Node.js версия Electron не поддерживает ReadableStream в fetch.
- Неполные JSON-строки при чанковатом чтении — нужно буферизировать строки.

**Решение**: Буферизация строк в `openrouterService.ts`: `buffer += decoder.decode(value)`, `lines = buffer.split('\n')`, `buffer = lines.pop()`. Каждая полная строка парсится как JSON. Неполная строка остаётся в буфере до следующего чанка.

**Версия**: V100
