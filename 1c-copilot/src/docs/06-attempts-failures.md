# 06 — Что НЕ сработало

## 1. Захват микрофона через `ffmpeg -f dshow` из Main-процесса

**Что пытались**: Использовать `child_process.spawn('ffmpeg', ['-f', 'dshow', '-i', 'audio=Микрофон (Realtek Audio)', ...])` для захвата микрофона в Main-процессе Electron.

**Почему не сработало**: Три фатальные проблемы:

1. **Кириллица в именах устройств**: На русскоязычных Windows устройства называются «Микрофон», «Стерео микшер» и т.д. При передаче этих строк в `child_process.spawn()` через аргументы, кодировка ломается: UTF-8 строки Node.js некорректно интерпретируются ffmpeg/DirectShow, которые ожидают ANSI/UTF-16. Результат — кракозябры вместо имён устройств, ffmpeg не находит устройство и падает.

2. **Нестабильный запуск dummy-устройств**: Попытка обойти проблему кириллицы через `audio_device_number` (индекс вместо имени) тоже ненадёжна: на разных машинах индексы разные, нулевое устройство может быть отключено, а попытка захвата несуществующего устройства блокирует инициализацию всего приложения.

3. **Ложный статус захвата**: `spawn()` возвращает процесс немедленно, и код ставил `captureState.mic = true` до реальной проверки что ffmpeg начал захватывать аудио. Если ffmpeg падал через 0.5с после старта — приложение считало что захват идёт.

**Решение**: Полный отказ от ffmpeg для микрофона. Перенос захвата в Renderer-процесс через `navigator.mediaDevices.getUserMedia({audio: true})`. Chromium сам управляет аудио-устройствами, корректно работает с любой локалью Windows, и предоставляет PCM-данные через AudioContext.

**Дата**: Июнь 2026 (сессия аудита и рефакторинга)

## 2. Рекурсивный ffmpeg fallback с dummy-устройствами

**Что пытались**: Реализовать цепочку fallback для ffmpeg dshow: сначала попробовать устройство по имени, потом по индексу, потом `audio_device_number 0`. Тип `CaptureCandidate`: `{ type: 'named', name } | { type: 'index', index } | { type: 'direct', args[] }`. Рекурсивная функция `tryNextDevice()`.

**Почему не сработало (полностью)**: Хотя рекурсивная логика была реализована корректно, фундаментальная проблема кодировки кириллицы осталась: даже при успешном захвате по индексу, ffmpeg мог захватить не тот девайс. На машине разработчика индекс 0 мог быть «Микрофон», а у пользователя — «Line In» или отключённое устройство. Dummy-захват `audio=dummy` — недокументированная функция DirectShow, работает непредсказуемо.

**Решение**: Вся стратегия ffmpeg для микрофона признана тупиковой. Миграция на getUserMedia.

**Дата**: Июнь 2026

## 3. Захват системного звука через Web Audio API

**Что пытались**: Использовать Web Audio API в renderer-процессе для захвата системного звука через `navigator.mediaDevices.getUserMedia()`.

**Почему не сработало**: Web Audio API не может захватить системный звук (loopback) — это ограничение безопасности браузера. `getUserMedia()` предоставляет доступ только к микрофону и камере. Для системного звука нужен нативный код.

**Решение**: Системный звук остаётся в Main-процессе через WASAPI loopback (`win-audio-capture`) с fallback на ffmpeg. Важно: WASAPI обёрнут в try/catch — если у пользователя нет стереомикшера или драйвер занят, приложение продолжает работать в режиме «только микрофон».

**Дата**: V100 (дизайн-решение, до реализации)

## 4. Отправка raw PCM на Whisper API

**Что пытались**: Отправлять raw PCM-данные (16kHz Mono 16-bit) напрямую на Whisper API, ожидая что API сам распознает формат.

**Почему не сработало**: Whisper API требует корректный контейнер (WAV, MP3, M4A, WEBM). Raw PCM без заголовка — невалидный аудио-файл, API возвращает ошибку формата (HTTP 400).

**Решение**: Функция `createWavBuffer()` в sttService.ts — добавляет 44-байтный WAV-заголовок (RIFF/WAVE/fmt/data) к PCM-данным перед отправкой.

**Дата**: V100

## 5. Использование npm-пакета `form-data` для multipart

**Что пытались**: Использовать популярный npm-пакет `form-data` для формирования multipart/form-data запросов к Whisper API.

**Почему не сработало**: Добавляет лишнюю зависимость, а в Electron main-процессе с `externalizeDepsPlugin()` нативные зависимости могут ломать сборку. К тому же `form-data` не всегда корректно работает с Buffer в Electron.

**Решение**: Ручная сборка multipart через `Buffer.concat()` — строки заголовков как Buffer.from(string, 'utf-8'), бинарные данные WAV как есть. Boundary генерируется динамически. Это работает и не требует зависимостей.

**Дата**: V100

## 6. Параллельная отправка чанков на STT

**Что пытались**: Отправлять чанки на Whisper API параллельно (Promise.all или несколько fetch одновременно) для снижения задержки.

**Почему не сработало**: Groq rate limit — 30 запросов/мин на бесплатном плане. Параллельная отправка 2-3 чанков одновременно может исчерпать лимит за секунды. Кроме того, порядок результатов не гарантируется — транскрипции могут прийти в неправильном порядке.

**Решение**: Последовательная обработка очереди (while loop в `processQueue()`). Следующий чанк отправляется только после получения результата предыдущего.

**Дата**: V100

## 7. ffmpeg dshow «Stereo Mix» на всех Windows-машинах

**Что пытались**: Использовать ffmpeg с dshow и устройством `audio=Stereo Mix (Realtek Audio)` как fallback для захвата системного звука на Windows.

**Почему не сработало**: «Stereo Mix» — виртуальное устройство, которое по умолчанию ОТКЛЮЧЕНО в настройках звука Windows. Многие аудиодрайверы (особенно не-Realtek) не предоставляют его вообще. Имя устройства различается на разных машинах.

**Решение**: Основной метод — WASAPI loopback через win-audio-capture (не зависит от Stereo Mix). ffmpeg fallback — запасной вариант, но он ненадёжен для системного звука на Windows.

**Дата**: V100

## 8. macOS системный звук через ffmpeg avfoundation

**Что пытались**: Использовать ffmpeg с `avfoundation` и устройством `:BlackHole 2ch` для захвата системного звука на macOS.

**Почему не сработало**: BlackHole — сторонний виртуальный аудио-драйвер, его нужно устанавливать отдельно. Без него avfoundation не может захватить системный звук. На Apple Silicon Mac есть дополнительные ограничения на доступ к аудио-устройствам.

**Решение**: ffmpeg fallback с BlackHole — документирован, но требует ручной установки пользователем. В будущем нужен нативный AudioUnit-based захват для macOS (coreaudio).

**Дата**: V100

## 9. Использование `fetch` с `response.body` в Electron для SSE

**Что пытались**: Использовать нативный `fetch()` в Electron main-процессе для чтения SSE-стрима от OpenRouter.

**Почему не сработало (частично)**: В целом это работает, но есть подводные камни:
- `response.body?.getReader()` может вернуть null если Node.js версия Electron не поддерживает ReadableStream в fetch.
- Неполные JSON-строки при чанковатом чтении — нужно буферизировать строки.

**Решение**: Буферизация строк в `openrouterService.ts`: `buffer += decoder.decode(value)`, `lines = buffer.split('\n')`, `buffer = lines.pop()`. Каждая полная строка парсится как JSON. Неполная строка остаётся в буфере до следующего чанка.

**Дата**: V100
