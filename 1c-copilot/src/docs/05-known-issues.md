# 05 — Известные проблемы и техдолг

## Критические проблемы

### 1. win-audio-capture нативный модуль — потенциальная проблема сборки

**Симптом**: `win-audio-capture` — нативный C++ модуль. При сборке electron-vite может не скомпилироваться если нет правильной версии node-gyp / Visual Studio Build Tools. Модуль добавлен в `dependencies`, но НЕ в `externalizeDepsPlugin` — это значит он может быть обработан Vite bundler'ом, что сломает нативный require.

**Почему критично**: Без нативного модуля WASAPI loopback не работает, fallback на ffmpeg может не захватить системный звук правильно.

**Обход**: В audioCapture.ts используется try/catch при require('win-audio-capture'), и fallback на ffmpeg. Но ffmpeg тоже нужно установить в системе.

**Статус**: Открыто

---

### 2. ffmpeg fallback — зависимость от внешнего процесса

**Симптом**: На macOS и Linux аудио-захват требует установленный ffmpeg в PATH. На Windows ffmpeg fallback использует dshow с `audio=Stereo Mix (Realtek Audio)` — имя устройства захардкожено и может отличаться на другой машине.

**Почему критично**: На большинстве машин Stereo Mix отключён в настройках звука Windows. Без него ffmpeg не захватит системный звук.

**Статус**: Открыто

---

### 3. Rate limit Groq при интенсивном использовании

**Симптом**: Groq бесплатный план — 30 запросов/мин. При двух потоках (mic + system) с чанками по 6с — до 20 запросов/мин. Если использовать приложение непрерывно больше 3 минут, можно упереться в лимит.

**Почему критично**: STT-очередь начнёт накапливаться, задержка транскрипции растёт.

**Обход**: Можно увеличить CHUNK_DURATION_MS до 10с (снизит до ~12 запросов/мин). Или переключиться на OpenAI (платный, но без жёсткого лимита).

**Статус**: Открыто

## Проблемы конкретных модулей

### audioCapture.ts

- **Нет VAD (Voice Activity Detection)**: все чанки отправляются на STT, включая тишину. Whisper возвращает пустой текст для тишины, но API-вызов всё равно тратится. Нужно добавить обнаружение голосовой активности.
- **ffmpeg health-check**: проверяет `ffmpeg.killed` каждые 5с, но не перезапускает процесс автоматически. Если ffmpeg падает — захват прекращается молча.
- **WASAPI sample rate захардкожен**: предполагается 44.1kHz стерео от WASAPI. На некоторых системах может быть 48kHz или 96kHz — резэмплирование будет некорректным.
- **ChunkBuffer.flush()**: при остановке захвата оставшиеся данные отправляются как неполный чанк. Если данных мало — Whisper может вернуть пустой результат.

### sttService.ts

- **Последовательная обработка**: чанки обрабатываются по одному (не параллельно). Это хорошо для rate limit, но увеличивает задержку. При большом потоке чанки накапливаются в очереди.
- **Нет retry**: при ошибке API чанк просто пропускается. Нет механизма повторной отправки.
- **Нет дедупликации**: если sttQueue разрастается, а транскрипции приходят с задержкой, текст может быть неактуален.
- **broadcastTranscription отправляет дважды**: сначала через getWindow(), потом через BrowserWindow.getAllWindows(). Это дублирует отправку в одно и то же окно.

### openrouterService.ts

- **Нет exponential backoff**: при ошибке API нет повторной отправки. Пользователь должен нажать «Спросить ИИ» вручную.
- **Системный промпт зашит в код**: нельзя изменить без пересборки. Нужно вынести в настройки или редактор промптов.
- **max_tokens: 1024**: для длинных подсказок может не хватить. Нет индикации обрезки.
- **temperature: 0.4 захардкожен**: нельзя изменить через UI.

### handlers.ts

- **cleanup() не вызывается**: функция определена, но не вызывается при `app.on('before-quit')`. Аудио-захват и стриминг могут не остановиться корректно.
- **IPC `transcription.update`**: зарегистрирован через `ipcMain.on` (fire-and-forget), но также отправляется из main через `broadcast`. Это может конфликтовать.

### overlayWindow.ts

- **Позиция окон**: при запуске все окна центрируются по верхнему краю экрана. Если разрешение меняется — окна не адаптируются.
- **maxHeight: 800**: захардкожен. Для transcript-панели с длинной расшифровкой может не хватить.

### SettingsPanel.tsx

- **Пошаговое сохранение**: каждая настройка сохраняется отдельным `settings.save()` вызовом. Если приложение упадёт в середине — часть настроек сохранится, часть нет.
- **Нет валидации**: API ключи не проверяются при сохранении. Пользователь может ввести некорректный ключ.

## Технический долг

1. **Нет тестов**: ни unit, ни integration. Критичные сервисы (audioCapture, sttService, openrouterService) не покрыты тестами.
2. **Нет логирования в файл**: только console.log. При багах в продакшене нет возможности посмотреть логи.
3. **Нет упаковки**: нет electron-builder / electron-forge конфигурации. Приложение нельзя установить как .exe.
4. **Стили inline**: CSS определён внутри компонентов через `<style>{`...`}</style>`. Нет CSS modules, нет Tailwind. Дублирование стилей между компонентами.
5. **Нет TypeScript strict**: tsconfig не включает `strict: true`. Могут быть неявные any и null-ошибки.
6. **pkg/ директория**: содержит копию package.json без win-audio-capture. Два package.json — источник путаницы.
7. **PROJECT_BRAIN.md пустой**: в pkg/PROJECT_BRAIN.md нет содержимого. Информация о проекте не задокументирована в оригинальном файле.
8. **Нет .env / dotenv**: API ключи хранятся в electron-store, но нет механизма для разработки с .env файлом.
9. **Нет Markdown-рендеринга**: SuggestionPanel отображает Markdown как plain text. Нужен react-markdown или аналог.
10. **Нет hotkey**: нет глобальных горячих клавиш для управления записью.
