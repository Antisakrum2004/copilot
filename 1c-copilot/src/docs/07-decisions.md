# 07 — Решения и обоснования

## 1. Перенос микрофона в Renderer-процесс через getUserMedia

**Что решили**: Захват микрофона реализовать в Renderer-процессе через `navigator.mediaDevices.getUserMedia({audio: true})` с AudioContext (PCM 16kHz, 1 канал, 16-бит) и пересылкой чанков через IPC. Отказаться от захвата микрофона через `ffmpeg -f dshow` в Main-процессе.

**Обоснование**:
- **Изоляция от кодировок Windows**: Chromium сам управляет аудио-устройствами и не зависит от проблем кодировки кириллицы в именах устройств. Нет кракозябр, нет зависимости от локали Windows.
- **Нативный и безопасный доступ**: getUserMedia — стандартный Web API, поддерживаемый Chromium/Electron. Не требует внешних зависимостей (ffmpeg в PATH, DirectShow драйверы).
- **Стабильность**: ffmpeg процесс мог упасть через 0.5с после spawn, и приложение считало захват активным. getUserMedia — persistent stream, который живёт пока жив Renderer.
- **Стандартизация**: Web Audio API даёт предсказуемый формат PCM через AudioContext с заданным sampleRate.

**Альтернативы**:
- ffmpeg dshow — кракозябры, нестабильный spawn, dummy-устройства
- PortAudio нативный модуль — сложнее сборка, не работает на Linux/macOS
- N-API нативный модуль — overkill для простого захвата микрофона

**Дата**: Июнь 2026

## 2. Изоляция WASAPI системного звука в try/catch

**Что решили**: Обернуть весь код WASAPI (win-audio-capture) в try/catch блоки. Если у пользователя нет стереомикшера, драйвер занят, или win-audio-capture не загружается — приложение должно продолжать работу в режиме «только микрофон», а не падать при старте.

**Обоснование**:
- WASAPI loopback зависит от наличия стереомикшера и корректного аудио-драйвера. На виртуальных машинах, бездрайверных аудиокартах, или при занятом устройстве — WASAPI не работает.
- Падение при старте из-за отсутствия системного звука недопустимо — микрофонный режим полезен сам по себе.
- try/catch обеспечивает graceful degradation: приложение запускается, пользователь видит что системный звук недоступен, но может работать с микрофоном.

**Альтернативы**:
- Фатальная ошибка при отсутствии системного звука — неприемлемо
- Пропуск без логирования — пользователь не понимает почему нет подсказок

**Дата**: Июнь 2026

## 3. Electron вместо web-приложения

**Что решили**: Реализовать 1C-Copilot как десктопное Electron-приложение, а не web-приложение в браузере.

**Обоснование**: Нужен доступ к системному аудио (WASAPI loopback) и overlay поверх всех окон (alwaysOnTop). В браузере невозможно захватить системный звук или создать окно поверх других приложений. Electron даёт доступ к нативным API через Node.js и возможность создавать frameless transparent окна.

**Альтернативы**:
- Web-приложение с Web Speech API — нет доступа к системному звуку, нет overlay
- Tauri — легче, но меньше экосистема, сложнее с нативными модулями (win-audio-capture)
- Чистое C++/Qt — мощнее, но значительно сложнее разработка UI

**Дата**: Начало проекта

## 4. Три отдельных overlay-окна вместо одного

**Что решили**: Создать 3 отдельных BrowserWindow: toolbar, suggestion, transcript — вместо одного большого окна.

**Обоснование**: Раздельные окна позволяют независимо управлять видимостью, прозрачностью, позицией и размером каждой панели. Тулбар всегда виден вверху, подсказки — сбоку, транскрипция — по желанию. Пользователь может закрыть транскрипцию, не закрывая подсказки. Каждое окно — отдельный renderer-процесс, что улучшает стабильность (краш одного окна не убивает другие).

**Альтернативы**:
- Одно окно с вкладками — нельзя расположить панели независимо
- Одно окно с layout-менеджером — сложнее реализовать drag-and-drop

**Дата**: Начало проекта

## 5. Hash-based роутинг вместо отдельных HTML-файлов

**Что решили**: Использовать один index.html с hash-роутингом (`#/toolbar`, `#/suggestion`, `#/transcript`) вместо трёх отдельных HTML-файлов.

**Обоснование**: Electron загружает один и тот же renderer bundle, а окна различаются по hash. Это упрощает сборку (один HTML entry point) и позволяет делить код между окнами (React-компоненты, стили, типы). Каждое окно создаётся с `win.loadURL(url#/<hash>)`.

**Альтернативы**:
- Три HTML entry point — сложнее сборка, дублирование bundle
- Query params — менее семантично, сложнее парсить

**Дата**: Начало проекта

## 6. Чанки по 6 секунд

**Что решили**: Нарезать аудио на чанки по 6 секунд (96 000 сэмплов при 16kHz, 192 000 байт).

**Обоснование**: 5-7 секунд — оптимальный размер для Whisper API. Короче 3 секунд — плохое качество распознавания. Длиннее 10 секунд — задержка транскрипции. 6 секунд — баланс между качеством и задержкой. При двух потоках (mic + system) — до 20 чанков/мин, что в пределах rate limit Groq (30/мин).

**Альтернативы**:
- 3 секунды — слишком короткие, хуже качество, больше API-вызовов
- 10 секунд — больше задержка, но меньше запросов
- VAD-based (переменная длина) — идеальный вариант, но сложнее реализовать

**Дата**: Начало проекта

## 7. Ручная сборка multipart вместо npm form-data

**Что решили**: Формировать multipart/form-data вручную через Buffer.concat(), без npm-зависимости form-data.

**Обоснование**: Electron main-процесс с externalizeDepsPlugin() может ломать нативные зависимости. form-data — лишняя зависимость для простого запроса с 4 полями. Ручная сборка даёт полный контроль и не добавляет уязвимостей.

**Дата**: Начало проекта

---

## 8. Последовательная STT-очередь вместо параллельной

**Что решили**: Обрабатывать чанки STT по одному (последовательно), не параллельно.

**Обоснование**: 1) Rate limit Groq — 30 запросов/мин. Параллельная отправка исчерпает лимит. 2) Порядок результатов гарантирован — транскрипции идут в хронологическом порядке. 3) Проще обрабатывать ошибки — не нужен сложный retry-механизм.

**Дата**: Начало проекта

## 9. qwen/qwen-2.5-coder-32b-instruct как модель по умолчанию

**Что решили**: Использовать `qwen/qwen-2.5-coder-32b-instruct` как модель LLM по умолчанию.

**Обоснование**: Qwen 2.5 Coder — дешёвая модель, хорошо работает с кодом и техническими подсказками. claude-3.5-sonnet точнее, но значительно дороже и медленнее. Для кратких 1С-подсказок с max_tokens=1024 Qwen достаточна. Пользователь может переключиться на другую модель в настройках.

**Дата**: Начало проекта

## 10. Debug-режим для overlay-окон

**Что решили**: В текущей версии (V1.0.0) suggestion и transcript окна создаются непрозрачными с тёмным фоном `#14141e` и рамкой (`frame: true`). Это временный режим для отладки.

**Обоснование**: На headless-сервере невозможно проверить работу прозрачных overlay-окон. При переносе на Windows-машину с GUI для отладки нужны видимые окна. В продакшен-режиме окна вернутся к `transparent: true`, `focusable: false`, `#02000000` фону и click-through логике.

**Альтернативы**:
- Оставить прозрачные — не видно на сервере, невозможно отладить
- Только DevTools — недостаточно для проверки overlay-позиционирования

**Дата**: Июнь 2026

## 11. electron-store вместо файлов/базы данных

**Что решили**: Использовать electron-store для хранения настроек (API ключи, модель, прозрачность).

**Обоснование**: electron-store — простая обёртка над JSON-файлом в userData директории. Автоматически создаёт файл, поддерживает схемы, типобезопасный API. Для 6 настроек этого достаточно — не нужна база данных.

**Дата**: Начало проекта

---

## 12. Material Dark + glass-morphism вместо Tailwind

**Что решили**: Использовать CSS Variables с Material Dark палитрой и glass-morphism эффектами (backdrop-filter: blur) вместо Tailwind CSS.

**Обоснование**: Overlay-приложение требует полупрозрачности и glass-эффектов. Tailwind — отличная система, но для overlay с custom backdrop-filter и RGBA-переменными CSS Variables дают больше контроля. Меньше зависимостей, проще отлаживать.

**Дата**: Начало проекта
